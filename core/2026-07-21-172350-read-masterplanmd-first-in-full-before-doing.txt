● Good catches — I need to adjust the plan on two points:

  1. Process spawn must not happen on a tokio worker thread. PR_SET_PDEATHSIG is
  per-thread; if the model's parent thread gets recycled by tokio, the child
  inherits a stale signal disposition. I should either spawn the process
  synchronously (before entering an async runtime) or use
  tokio::task::spawn_blocking to ensure the fork/exec happens on a dedicated OS

● I've analyzed all four bugs. Let me fix them one by one.

  Bug 1: Config::load() uses manual TOML Value merging that only merges the
  global section — schema_version and models from the parsed file are discarded.
  Fix: use toml::from_str::<Config>(&content) directly, relying on serde's
  #[serde(default)] for missing fields (the real defaults — log_dir →
  ~/.local/share/ai-switch/logs/, proxy_port → 8080,
  auto_restart_on_context_full → true — are applied in the accessor methods).

  Bug 2: SingleInstanceGuard is dropped when run() returns. Fix: return it as
  part of run()'s result so the caller keeps it alive.

  Bug 3: LinuxProcessGuard::setup doesn't check if the child already exited
  after fork. Fix: return Err(ProcessError::Spawn(...)) in that case.

  Bug 4: Race between prctl(PR_SET_PDEATHSIG) and parent death. Fix: compare
  getppid() against original parent pid after setting PDEATHSIG.
